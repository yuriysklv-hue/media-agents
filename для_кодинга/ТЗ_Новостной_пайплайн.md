# ТЗ: media-agents — новостной пайплайн

> **Документ для передачи в Claude Code.** На основе этого файла реализуется репозиторий `media-agents` — пайплайн сбора, перевода, рерайта и публикации западных AdTech-новостей.

***

## 1. Что строим

### 1.1. Scope MVP

**Входит:**

* Сбор новостей из 5 валидированных западных RSS-фидов (Adweek, AdExchanger, Digiday, MediaPost TV, MediaPost Media).
* Полный конвейер: Collector → Pre-Filter → Translator → Filter+Dedup → Author News → Enricher → QA → Publisher.
* **Осмысленный рерайт** — не перевод, а оригинальная новость на русском, написанная журналистским языком по фактам из источника.
* Недельный дайджест по всем опубликованным новостям со ссылками на них.
* Публикация через **Git PR** в репозиторий `media-site`.
* Запуск **локально** (ручная команда) и **на GitHub Actions** (по расписанию).

**НЕ входит (на следующих итерациях):**

* Коллекторы press-release, vendor-blog, earnings.
* Источники Россия/CIS (MVP-2) и Азия (исключена).
* Типы контента: review, report, column.
* Telegram-бот для алертов.
* Эскалация на Claude Sonnet/Opus (инфраструктура заложена, но не активируется).

### 1.2. Архитектурные принципы (из `03_Архитектура_агентов.md`)

1. **Конвейер, а не один агент.** Каждый шаг — отдельный модуль.
2. **Перевод — промежуточный этап.** Translator даёт рабочий перевод для извлечения фактов. Author News пишет оригинальный текст с нуля. Перевод в финальный файл не попадает.
3. **Markdown — lingua franca.** Все агенты производят .md файлы со стандартизованным front-matter.
4. **Идемпотентность.** Повторный запуск не плодит дубли.
5. **Многоуровневая LLM-стратегия.** DeepSeek — перевод и написание; GLM — фильтрация, enricher, QA, embeddings.

### 1.3. Ссылки на контекст

* `03_Архитектура_агентов.md` — полная архитектура (все типы контента, все коллекторы).
* `00_Позиционирование_медиа.md`, раздел 7 — принцип «анти-ИИ-голос».
* `06_SEO_GEO_и_AI-разметка.md` — правила SEO, slug-генерации, чек-лист публикации.

## 5. Схемы данных

### 5.1. raw\_item — на выходе из Collector

```json
{
  "id": "sha256(source_url)[:16]",
  "source_type": "rss",
  "source_name": "AdExchanger",
  "source_url": "https://www.adexchanger.com/...",
  "title": "Original English Title",
  "summary": "RSS summary / description field",
  "content": "Full article text extracted via trafilatura",
  "content_extracted": true,
  "language": "en",
  "region": "world",
  "published_at": "2026-07-06T10:30:00Z",
  "collected_at": "2026-07-06T12:00:00Z"
}
```

Поля:

* `id` — хеш от `source_url`, детерминированный.
* `content_extracted` — `false`, если trafilatura не смог извлечь полный текст (тогда `content` \= `summary`).
* `published_at` — из RSS `<pubDate>`, нормализованный в ISO 8601 UTC.

### 5.2. passed\_pre\_filter\_item — на выходе из Pre-Filter

Наследует все поля `raw_item` + добавляет:

```json
{
  "...all raw_item fields...": "",
  "keyword_score": 35,
  "keyword_matched": ["programmatic", "DSP"],
  "llm_score": 8,
  "llm_reason": "Прямая новость о запуске DSP-функциональности",
  "pre_filter_passed": true,
  "pre_filter_at": "2026-07-06T12:01:00Z"
}
```

### 5.3. translated\_item — на выходе из Translator

Наследует все поля + добавляет переведённые:

```json
{
  "...all fields from pre_filter...": "",
  "title_ru": "Переведённый заголовок",
  "summary_ru": "Переведённое саммари",
  "content_ru": "Переведённый полный текст",
  "translated_at": "2026-07-06T12:02:00Z",
  "translator_model": "deepseek-chat"
}
```

> Переведённые поля — **рабочий материал**, не для публикации. Author News пишет оригинальный текст.

### 5.4. curated\_item — на выходе из Filter+Dedup

```json
{
  "event_id": "sha256(best_source_url)[:16]",
  "created_at": "2026-07-06T12:05:00Z",
  "cluster_topic": "Краткое описание события (сгенерировано LLM)",
  "relevance_score": 8.5,
  "sources": [
    {
      "source_name": "AdExchanger",
      "source_url": "https://www.adexchanger.com/...",
      "title_original": "Original English Title",
      "title_ru": "Переведённый заголовок",
      "summary_ru": "Переведённое саммари",
      "content_ru": "Переведённый полный текст",
      "individual_score": 8.5,
      "is_primary": true
    }
  ],
  "source_count": 1,
  "is_duplicate": false,
  "duplicate_of": null
}
```

Если несколько источников покрывают одно событие — все попадают в `sources[]`. Author News синтезирует их в одну статью. `is_primary: true` у лучшего источника (по релевантности и полноте).

### 5.5. news .md — финальный файл новости

Файл: `media-site/src/content/news/{slug}.md` (в репо `yuriysklv-hue/media`)

URL на сайте: `/article/{slug}/`

```yaml
---
title: "Заголовок на русском (50-80 символов)"
description: "Описание для SEO, до 160 символов"
pubDate: 2026-07-06T12:00:00Z
author: news-world            # reference → authors collection
category: adtech-world
geo:
  - МИР                       # РФ | МИР | АЗИЯ
tags:
  - programmatic
  - DSP
featured: false               # true только для героя главной (руками)
readingTime: 3                # минуты, расчёт в Enricher
highlights: []                # зарезервировано (пустой массив для новостей)
source:
  title: AdExchanger
  url: https://www.adexchanger.com/...
---

Текст новости на русском (оригинальный рерайт, не перевод).
Структура: лид → суть (2-4 абзаца) → что это значит → ссылка на источник.
```

> **Поле `type` во front-matter НЕ нужно.** Тип материала = имя коллекции в Astro Content Collections (`news`, `digest`, `reviews`, `columns`, `reports`). Файл в `src/content/news/` — уже новость. Единый URL для всех материалов: `/article/{slug}/`.

### 5.6. digest .md — финальный файл дайджеста

Файл: `media-site/src/content/digest/{slug}.md` (в репо `yuriysklv-hue/media`)

URL на сайте: `/article/{slug}/`

```yaml
---
title: "Дайджест недели N: главные события adtech"
description: "Краткое описание недели, до 160 символов"
pubDate: 2026-07-13T09:00:00Z
author: news-world            # reference → authors collection
category: adtech-world
geo:
  - МИР                       # РФ | МИР | АЗИЯ
tags:
  - weekly-digest
featured: false
readingTime: 7                # минуты, расчёт в Enricher
highlights:                   # ключевые тезисы для карточки дайджеста
  - "Главное событие недели"
  - "Тренд: усиление retail media"
  - "M&A: покупка X компанией Y"
sources_count: 12
week: "2026-W28"
---

Структура (см. промпт digest_writer.md):
1. Главная новость недели
2. В фокусе (2-3 новости)
3. Кратко (5-8 новостей по темам)
4. Что это значит для рынка
```

### 7.4. Filter + Dedup — `src/processors/filter_dedup.py`

**Назначение:** Финальная фильтрация на русском + семантическая дедупликация + кластеризация.

**Вход:** `data/inbox/translated_items.jsonl`.

**Выход:** `data/inbox/curated_items.jsonl`.

**Шаг 1 — Повторная фильтрация (GLM-4-Flash):**

Перевод может выявить нерелевантность, неочевидную на английском. Тот же промпт, что и Pre-Filter, но на русском тексте. Threshold: score ≥ 5.

**Шаг 2 — Семантическая дедупликация:**

1. Для каждого `translated_item` вычислить embedding через GLM `embedding-3`.
2. Загрузить embeddings опубликованных материалов из `data/published/embeddings.npz`.
3. Вычислить cosine similarity с каждым опубликованным.
4. Если max similarity > 0.85 → пометить как дубликат, убрать.

**Шаг 3 — Кластеризация новых:**

1. Вычислить pairwise cosine similarity между новыми `translated_items`.
2. Группировать items с similarity > 0.75 в кластеры (одно событие, несколько источников).
3. Для каждого кластера:
   * Выбрать primary source (максимальный relevance\_score, максимальная длина content).
   * Сформировать `curated_item` (схема 5.4) со всеми источниками.
4. LLM (GLM-4-Flash) генерирует `cluster_topic` — краткое описание события (1 предложение).

**Особенности:**

* Embeddings новых items сохраняются в `data/published/embeddings.npz` после публикации (для будущей дедупликации).
* Порог 0.85 для дедупликации против опубликованного, 0.75 для кластеризации новых.

**Slug generation** (`src/utils/slug.py`):

```python
from slugify import slugify

def generate_slug(title: str, max_length: int = 60) -> str:
    """Транслит + kebab-case + обрезка."""
    s = slugify(title, lowercase=True, separator='-')
    # Обрезка по границе слова
    if len(s) > max_length:
        s = s[:max_length].rsplit('-', 1)[0]
    return s
```

2. Создать ветку: `news-batch-{YYYYMMDD-HHMM}`.

   * Скопировать в `media-clone/media-site/src/content/news/{slug}.md`.
4. Commit: `news: add N articles [batch YYYYMMDD-HHMM]`.
5. Push ветку.
6. Создать PR через `gh pr create`:

```
## Новости — batch {timestamp}

{N} статей прошло полный пайплайн.

### Статьи

| # | Заголовок | Slug | Источник | QA |
|---|-----------|------|----------|----|
| 1 | {title} | {slug} | {source_name} | PASS |
| ... | | | | |

### Метрики

- Собрано raw_items: {N}
- Прошло pre-filter: {N}
- Переведено: {N}
- Прошло filter+dedup: {N}
- QA FAIL: {N}
- LLM cost: ${cost}
- Время выполнения: {duration}
```

1. Записать в `published.jsonl` для каждого файла (схема 5.7).
2. Обновить `embeddings.npz` — добавить embeddings новых статей.
3. Очистить `data/drafts/news/`.

**Особенности:**

* В DRY\_RUN режиме — шаги 1-5 пропускаются, PR не создаётся. Файлы остаются в `drafts/`.
* Если GH\_TOKEN не имеет прав — fail с понятным сообщением.
* Размер batch: все QA PASS статьи за один прогон пайплайна (обычно 3-8 статей).

### 7.9. Author Digest — `src/writers/digest_writer.py`

**Назначение:** Недельный дайджест по всем опубликованным новостям со ссылками на них.

**Вход:** записи из `published.jsonl` за последние 7 дней (или указанная неделя).

**Выход:** .md файл в `data/drafts/digest/{slug}.md`.

**Модель:** DeepSeek-V3 (`deepseek-chat`).

**Промпт** (`src/prompts/digest_writer.md`):

```

по новостям недели.

Структура:

1. **Главная новость недели** — одно событие, которое выходит за рамки

   Ссылка на материал: [{title}](/article/{slug})

2. **В фокусе** — 2-3 значимые новости, каждая с коротким комментарием
   (1-2 абзаца). Ссылка на каждый материал.

3. **Кратко** — 5-8 новостей, сгруппированных по темам:
   - Retail Media
   - AI & Automation
   - Measurement & Privacy
   - Платформы (Google, Meta (запрещён в РФ), Amazon, ...)
   - M&A и инвестиции
   Каждая — заголовок + 1 предложение + ссылка: [{title}](/article/{slug})

4. **Что это значит для рынка** — 2-3 наблюдения с оговорками
   («возможно», «при определённых условиях», «пока рано судить»).

Объём: 800-1500 слов. Тон: экспертно-аналитический, журналистский.
Короткие абзацы по 1-2 предложения. Допустима лёгкая ирония.
Меньше глобальных выводов — наблюдения, а не прогнозы.

ПРИНЦИП «АНТИ-ИИ-ГОЛОС» — ОБЯЗАТЕЛЬНО.
Без кликбейта, без «революций» и «переписывания правил».

{news_list_with_slugs}

Верни результат с front-matter type: digest.
```

**Подстановка `{news_list_with_slugs}`:**

## 8. Pipeline orchestration

### run\_pipeline.py — главный CLI

```python
"""
Главный скрипт пайплайна новостей.

Usage:
  python run_pipeline.py                    # Полный цикл
  python run_pipeline.py --collect-only     # Только сбор
  python run_pipeline.py --skip-collect     # Обработка существующих raw_items
  python run_pipeline.py --dry-run          # Без git-операций
  python run_pipeline.py --stage translator # Запуск только одного этапа
"""
```

**Полный flow:**

```
1. RSS Collector          → raw_items.jsonl
2. Pre-Filter             → passed_pre_filter.jsonl
3. Translator             → translated_items.jsonl
4. Filter + Dedup         → curated_items.jsonl
5. Для каждого curated_item:
   a. Author News         → drafts/news/{slug}.md
   b. Enricher            → обновлённый front-matter
   c. QA                  → PASS | FAIL
6. Собрать все PASS файлы
7. Publisher              → PR в media-site
8. Обновить state-файлы
```

**Логирование:**

* Каждый этап пишет начало/конец в `data/logs/runs.log`.
* `data/state/last_run.json` — timestamps последнего запуска каждого этапа.
* `data/state/llm_usage.jsonl` — каждый LLM-вызов.

### run\_digest.py — CLI дайджеста

```python
"""
Дайджест недели.

Usage:
  python run_digest.py                  # Текущая неделя
  python run_digest.py --week 2026-W27  # Конкретная неделя
  python run_digest.py --dry-run        # Без PR
"""
```

**Flow:**

```
1. Загрузить published.jsonl за неделю
2. Digest Writer (DeepSeek-V3) → drafts/digest/{slug}.md
3. Enricher → front-matter
4. QA → PASS | FAIL
5. Publisher → PR в media-site (ветка digest-{YYYY-WW})
```

## 10. Обработка ошибок

| Случай                                | Поведение                                                                                                                    |
| ------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------- |
| RSS-фид недоступен / timeout          | Пропуск с warning, не падать                                                                                                 |
| trafilatura не извлёк контент         | Использовать `summary` как `content`, отметить `content_extracted: false`                                                    |
| Превышение rate limit API             | Retry (3 попытки, экспоненциальная задержка: 2s → 4s → 8s)                                                                   |
| DeepSeek API недоступен               | Пауза пайплайна, алерт в лог. Не fallback на Claude в MVP                                                                    |
| GLM API недоступен                    | Pre-filter пропускается (только keyword scoring), enricher пропускается, QA — только rules-based. Pipeline продолжает работу |
| LLM вернул невалидный JSON            | Retry с промптом «верни строго валидный JSON». После 2 неудач — skip item, в `drafts/failed/`                                |
| LLM вернул невалидный Markdown / YAML | QA FAIL, черновик в `drafts/failed/`                                                                                         |
| Git push отклонён / PR не создан      | Откат, файлы остаются в `drafts/`, error в лог                                                                               |
| Embedding API недоступен              | Дедупликация пропускается (только URL-based), warning в лог                                                                  |
| Все items отброшены на pre-filter     | Нормальный сценарий, пустой PR не создаётся                                                                                  |

```python
class StateManager:
    """Управление персистентным состоянием пайплайна."""

                      input_tokens: int, output_tokens: int, cost_usd: float,
                      item_id: str = None):
```

## 14. Получение Zhipu API ключа (GLM-4-Flash)

GLM-4-Flash работает на бесплатном тарифе Zhipu AI, но требует регистрации.

1. Перейти на [open.bigmodel.cn](https://open.bigmodel.cn).
2. Зарегистрироваться (поддерживается вход через Google).
3. Раздел «API Keys» → «Create API Key».
4. Скопировать ключ в `.env`:
   ```
   ZHIPU_API_KEY=your_key_here
   ```
5. Бесплатный лимит: достаточно для MVP (QPS и токены/день).

> GLM-4-Flash бесплатен, но имеет лимиты на количество запросов в минуту (QPM). Pipeline отправляет пачками по 50 items с задержкой — лимиты не превышаются.
