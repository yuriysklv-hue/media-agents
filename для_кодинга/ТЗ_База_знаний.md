## 1. Контекст

Читатель 1screen встречает термины (CTV, MFA, first-party data), компании (Criteo, Magnite),
технологии (Advantage+, Conversions API) без быстрой справки. Решение — evergreen-раздел
**«База знаний»** по URL `/spravochnik/`: структурированный справочник, адаптированный под
СНГ-читателя (российские платформы в примерах, русские формулировки). Материалы генерятся
из **ручной очереди**, проходят фактчек + rules-QA, публикуются через **Git PR с ручным мержем**.

Отличия от новостного пайплайна: триггер — очередь (не RSS); контент evergreen/обновляемый
(не одноразовый); тон энциклопедический; нет collect/translate/dedup; есть Researcher и
feedback-loop (revision).

## 2. Типы контента

4 типа. Каждый — свой набор полей `facts` и свой Schema.org `@type` (генерится в Astro).
**`schema_jsonld` во front-matter НЕ пишется** (решение A).

### 2.1. `company` → Schema.org `Organization`
`facts`: `founded`(да), `founders[]`(да), `hq`(да), `parent_organization`, `ticker`,
`official_url`(да), `key_products[]`, `subtype`(platform/agency/broadcaster/media).
Body: `## О компании` → `## История` → `## Роль в рекламной индустрии` → `## Ключевые продукты`.

### 2.2. `technology` → `SoftwareApplication`
`facts`: `developer`(да), `launch_date`, `category`(да; AI campaigns/measurement/CTV ads…),
`pricing_model`, `alternatives[]`, `official_url`(да).
Body: `## Что это` → `## Как работает` → `## Для кого` → `## Альтернативы`.

### 2.3. `term` → `DefinedTerm`
`facts`: `category`(да; media_format/metric/methodology/concept), `definition`(да), `aliases[]`.
Body: `## Определение` → `## Контекст` → `## Примеры` (обязательно — российские платформы, анти-thin-content).

### 2.4. `organization` → `Organization` (ассоциация/стандарт)
`facts`: `full_name`(да), `founded`(да), `mission`(да), `official_url`(да), `key_initiatives[]`.
Body: `## Что это` → `## Ключевые инициативы` → `## Значимость`.

### Общие поля front-matter (все типы)
`title`, `description`(≤160), `type`, `slug`, `pubDate`(AUTO=utcnow), `updatedDate`(опц.),
`aliases[]`(опц.), `tags[]`(3–7, из словаря), `related[]`(опц.), `facts:{…}`, `author`(решение G).

## 3. Schema.org JSON-LD — генерируется в Astro

Слой Astro (`SpravochnikLayout.astro`/`[slug].astro`) детерминированно строит
`<script type="application/ld+json">` из `type` + `facts`:
- `company`/`organization` → `Organization` (+ `founders` как `Person[]`, `foundingDate`, `address`, `url`, `tickerSymbol`);
- `technology` → `SoftwareApplication` (`applicationCategory: AdvertisingApplication`, `creator`, `releaseDate`, `url`);
- `term` → `DefinedTerm` (`inDefinedTermSet: {name: "Справочник 1screen"}`).
Плюс `BreadcrumbList` (Главная → База знаний → {термин}). `dateModified = updatedDate ?? pubDate`
(как у новостей). **Валидность гарантируется кодом Astro, а не дисциплиной LLM.**

## 4. Архитектура пайплайна (backend, media-agents)

```
config/spravochnik_queue.yaml   ←── владелец добавляет термины (через Claude Code)
        │
        ▼  spravochnik.yml (Actions, cron каждые 6 ч + workflow_dispatch)
        │
python -m src.spravochnik.run
  0. Sync merged PRs   — GitHub API: status review→published, если PR смёржен
  1. Queue Manager     — следующий pending|revision (иначе выход)
  2. Researcher        — Wikipedia (normalize+timeout) + related из живой коллекции media
  3. Writer (DeepSeek) — шаблон по типу; тон энц.; + legal-сноска; feedback при revision
  4. Fact-checker(GLM) — сверка founded/founders/hq с Wikipedia (best-effort)
  4b. Rules-QA         — схема/длины/slug/кодировка/структура (детерминированно)
  5. Publisher         — reuse git_publisher: ветка spravochnik/{slug}, файл, draft-PR
     → queue: status review
```

Отдельный процесс (не в `run_pipeline.py`). Персист очереди — шаг «Commit queue state» в workflow.

## 5. Очередь `config/spravochnik_queue.yaml`

```yaml
# Статусы: pending → writing → review → published
#          ветки: revision (ручной reject), needs_fix (фактчек-блок), skipped
items:
  - id: adobe
    term: Adobe
    type: company
    subtype: platform
    status: pending
    iteration: 0
    feedback: null
    created_at: "2026-07-19"
    published_at: null
    slug: adobe
```

Поля по типам: `company`→`subtype`; `technology`→`developer`; `term`→`category`;
`organization`→`full_name`. `slug` — уникальный идентификатор (queue_manager проверяет при add).

**Lifecycle:**
```
pending ─writing→ review ─(merge)→ published
                    │
   revision ←(reject, владелец)┘
     │ feedback, iteration++
     └writing→ review → …
needs_fix ←(фактчек critical_error)  # владелец разбирает вручную
```

### Управление через Claude Code (админки нет)
- **Добавить:** дописать `pending`/`iteration:0`, проверить уникальность slug, коммит `spravochnik: queue +N items`.
- **На доработку:** закрыть PR в `media` без merge → `status: revision`, `iteration++`, `feedback: "…"`, коммит.
- **Статус:** прочитать YAML, вывести таблицу.
- **Удалить/пропустить:** убрать айтем или `status: skipped`.

### Функции `queue_manager.py`
```python
def load_queue() -> list[dict]
def save_queue(items: list[dict]) -> None
def get_next_item() -> dict | None          # первый pending|revision
def update_status(item_id, status, **fields) -> None
def add_item(term, type, **fields) -> dict   # slug-уникальность
```

## 7. Writer (DeepSeek-V3)

Промпт `src/prompts/spravochnik_writer.md`: энциклопедический нейтральный тон; факты
только из `research_bundle` (нет факта — не выдумывать, пометка `{{fact_check: "…"}}`);
структура строго по типу; adtech-термины без перевода (DSP/SSP/CPM/RTB); бренды в
устоявшемся русском написании; `description` **укладывается в ≤160 законченной фразой**
(решение G, расхождение примеров исправлено). Feedback-секция — только при revision.

Выход: front-matter (`facts` + общие поля, **без `schema_jsonld`**) + body по структуре типа.

**После генерации (детерминированно, `writer.py`, переиспользуя код новостей):**
1. `legal.add_restricted_org_footnotes(body)` — сноска РКН (решение B).
2. finalize-гейт: `_fit_description(≤160)`, валидация `tags` против `vocabulary`, `readingTime`,
   `pubDate = utcnow_iso()`, `author` (решение G), slug из очереди (уникальность через `ensure_unique`).

## 8. Fact-checker (GLM-4-Flash)

Промпт `spravochnik_fact_checker.md`. Сверяет `founded`/`founders`/`hq` с Wikipedia extract;
`description` — семантически. Возвращает `{passed, warnings[], critical_errors[]}`.
- `critical_errors` → `status: needs_fix`, `feedback = str(errors)` (владелец разбирает).
- Только `warnings` → проходит с пометкой.
- **`wikipedia == None` (technology/term) → фактчек пропускается** (не блокирует), warning
  «нет опорного источника — ручное ревью».

## 9. Rules-QA уровень 1 (`src/spravochnik/qa.py`)

По образцу `qa.check_rules`. FAIL при:
- отсутствии обязательных полей типа (`facts` по разделу 2);
- `description > 160`; `title` пустой/вне разумной длины;
- `type` вне {company,technology,term,organization}; `category` вне словаря типа;
- `slug` не уникален; `tags` вне словаря без `[new]`;
- **не-кириллица/латиница-мусор в теле** (иероглифы/битые куски — примеры ТЗ их содержали);
- битая структура заголовков (нет обязательных `## …` по типу);
- `pubDate` не ISO / в будущем.
Тело ≥ 500 символов (thin-content гард). FAIL → `data/spravochnik/failed/` + запись причины.

## 11. Workflow `spravochnik.yml` (Actions, решение F)

```yaml
on:
  schedule: [{cron: '0 */6 * * *'}]
  workflow_dispatch: {inputs: {dry_run: {type: boolean, default: false}}}
permissions:
  contents: write            # запись очереди (иначе 403, урок задачи 7)
jobs:
  run:
    steps:
      - run: python -m src.spravochnik.run
        env: {DEEPSEEK_API_KEY, ZHIPU_API_KEY, ZHIPU_BASE_URL, GLM_FLASH_MODEL,
               GH_TOKEN, MEDIA_REPO, MEDIA_SITE_SUBDIR}
      - name: Commit queue state    # config/spravochnik_queue.yaml → main
        if: ${{ !inputs.dry_run }}
```
`models.yaml`: добавить `spravochnik_writer: deepseek:chat`, `spravochnik_fact_checker: glm:flash`.
Throughput: 1 материал/прогон × 4/день. Стоимость ~$0.006/материал (~$0.72/мес). 45-мин лимит не грозит.
