# Архитектура агентов

> Главный документ для настройки пайплайна агентского контента. На основе него пишется код в репозитории `media-agents` (через Claude Code) и формулируются промпты в AI sreda.
>
> **Детализированное ТЗ для первого цикла (западные новости + дайджест):** `09_ТЗ_Новостной_пайплайн.md`. Этот документ — архитектурный обзор; ТЗ содержит точные схемы данных, сигнатуры модулей, конфигурацию и порядок реализации.

***

## 1. Принципы архитектуры

1. **Конвейер, а не один универсальный агент.** Каждый шаг выполняется отдельным специализированным агентом с узкой задачей. Это упрощает отладку, замену моделей и эволюцию.
2. **Человек в цикле только там, где это критично.** Короткие новости идут авто, аналитика и колонки — через ревью.
3. **Markdown — lingua franca.** Всё, что производят агенты, — это Markdown-файлы со стандартизованным фронт-маттером. Сайт ничего не знает про агентов.
4. **Идемпотентность.** Повторный запуск агента не должен плодить дубли. Дедупликация на нескольких уровнях.
5. **Минимум внешних зависимостей.** Git + LLM API + RSS. Никаких сложных очередей и брокеров на старте.
6. **Перевод — отдельная стадия, а не побочный эффект.** Промпт «автоматом переведи и напиши» даёт либо literal translation, либо галлюцинации. Перевод выделен в самостоятельного агента со своим промптом, обученным на рекламную терминологию.
7. **Многоуровневая LLM-стратегия (экономика).** Не всё пишем на Claude. Дешёвые модели — для массовых задач (фильтрация, перевод, enricher, короткие новости). Claude — только там, где качество критично и материалов мало (обзоры, отчёты, экспертная редактура).

***

## 3. Роли агентов

### 3.1. Коллекторы (Collectors) — `collectors/`

Не LLM-агенты, а Python-скрипты, которые ходят по источникам и пишут сырьё в `inbox/raw_items.jsonl`.

Базируются на коде из `/Kojori/adtechreview/session_2026_01_23_ideal_format/hybrid_generator.py` — коллектор RSS, алгоритм скоринга и дедупликация уже протестированы в production с января 2026.

| Коллектор                 | Источники                                 | Частота                     | Формат выхода |
| ------------------------- | ----------------------------------------- | --------------------------- | ------------- |
| `rss_collector`           | RSS-фиды (список ниже)                    | каждые 2 часа               | JSONL         |
| `press_release_collector` | Страницы пресс-релизов вендоров (парсинг) | раз в день                  | JSONL         |
| `vendor_blog_collector`   | Корпоративные блоги                       | раз в день                  | JSONL         |
| `earnings_collector`      | Financial reports (IR-страницы)           | раз в неделю / при триггере | JSONL         |

**Валидированные RSS-источники (из работающего бота):**

| Категория                                  | Источник                 | URL                                                 | Объём/день |
| ------------------------------------------ | ------------------------ | --------------------------------------------------- | ---------- |
| **Западный AdTech**                        | Adweek                   | `https://www.adweek.com/feed/`                      | 5-10       |
|                                            | AdExchanger              | `https://www.adexchanger.com/feed/`                 | 3-7        |
|                                            | Digiday                  | `https://digiday.com/feed/`                         | 2-5        |
|                                            | MediaPost TV             | `http://feeds.mediapost.com/television-news-daily/` | 2-4        |
|                                            | MediaPost Media          | `http://feeds.mediapost.com/mediadailynews/`        | 3-5        |
| **Россия/CIS** *(добавить на этапе MVP-2)* | Adindex                  | TBD                                                 | 5-10       |
|                                            | Sostav                   | TBD                                                 | 3-5        |
|                                            | VC.ru (раздел маркетинг) | TBD                                                 | 5-10       |

> **Азия/Pacific — исключены.** Источники (TechNode, Branding in Asia, MediaWeek AU, SCMP) тестировались в работе бота, качество материала недостаточное: много шума, слабая adtech-специфика, низкая плотность полезных новостей. Покрытие азиатского рынка (Alibaba, Tencent, Baidu) будет через пресс-релизы и блоги самих вендоров + периодический research-сбор Author Reports (см. 3.8).

**Скоринг-система (переносится из бота, используется в Pre-Filter):**

Четырёхуровневая система взвешенных ключевых слов, валидирована в production (67-85% эффективности фильтрации):

| Уровень                 | Вес в title / в desc | Примеры                                                                                                                          |
| ----------------------- | -------------------- | -------------------------------------------------------------------------------------------------------------------------------- |
| **CORE AdTech**         | 30 / 20              | programmatic, DSP, SSP, ad exchange, RTB, retail media, CTV, OTT, AI advertising, LLM, attribution, measurement, privacy, cookie |
| **PLATFORMS**           | 20 / 10              | Google, Meta (запрещён в РФ), Amazon, TikTok, Apple, Microsoft, OpenAI, Netflix, Reddit, Spotify, YouTube                        |
| **ADVERTISING GENERAL** | 10 / 5               | advertising, ads, advertisers, CMO, CPM, CTR, adtech, martech, media buying, analytics                                           |
| **GENERIC INDUSTRY**    | 5 / 3                | merger, acquisition, IPO, funding, layoff, CEO, earnings, revenue, stock                                                         |

**Пороги:** score ≥ 20 — проходит; score \< 20 — отбрасывается.
**Исключения (score \= 0):** sport ads, creative campaigns, crypto, gaming, news roundups (3+ semicolons в заголовке), print/billboard.
**Override:** сильные AdTech-ключевые слова (programmatic, DSP, SSP, retail media, CTV, AI advertising) перепроверяют исключения — если есть, материал проходит даже при срабатывании exclusion.

> **Источник:** `adtechreview/session_2026_01_23_ideal_format/hybrid_generator.py`, методы `calculate_relevance_score()`, `CORE_ADTECH_KEYWORDS`, `PLATFORM_KEYWORDS`, `EXCLUDED_TOPICS`.

**Дедупликация (на уровне коллектора):**

Извлечение article\_id из URL (regex `/article/(\d+)/`), хранение в множестве на время цикла. В новой архитектуре — персистентное хранение в `state/seen_urls.json` + SQLite для зрелого решения.

**Схема `raw_item`:**

```json
{
  "id": "hash",
  "source_type": "rss | press_release | blog | earnings",
  "source_name": "Digiday",
  "source_url": "https://digiday.com/...",
  "title": "...",
  "summary": "...",
  "content": "...",
  "language": "en",
  "published_at": "2026-07-03T10:00:00Z",
  "collected_at": "2026-07-03T12:00:00Z",
  "raw_html": "..."
}
```

### 3.2. Pre-Filter — `processors/pre_filter.py`

* **Модель:** GLM-4-Flash (бесплатный tier Zhipu API).
* **Вход:** поток `raw_item`-ов (пачками по 50–100).
* **Зачем нужен:** до перевода отбраковать явно нерелевантное (спорт, lifestyle, политика без рекламной темы). Не тратить токены DeepSeek на мусор.
* **Что делает:**
  1. Оценивает релевантность по title + summary по шкале 0–10.
  2. Отбрасывает всё со скором \< 5.
  3. Помечает оставшиеся скором для приоритизации.
* **Выход:** `inbox/passed_pre_filter.jsonl`.

**Промпт-каркас:**

```
Ты — фильтр информационного потока российского медиа о рекламе и adtech.
Оцени, относится ли заголовок и краткое описание к тематике медиа.

Тематика: реклама, маркетинг, adtech, рекламные технологии, маркетинговые
инструменты, M&A в adtech, регуляторика рекламы, креатив в рекламе,
рекламные продукты крупных платформ (Яндекс, VK, 2ГИС, Google, Meta (запрещён в РФ),
Alibaba, Baidu и т.д.).

Шкала:
- 0-3: явно нерелевантно.
- 4-6: сомнительно (упоминается бренд, но не про рекламу).
- 7-10: релевантно.

Верни строго JSON: {"score": 7, "reason": "короткое объяснение"}
```

### 3.3. Translator — `processors/translator.py`

* **Модель:** **DeepSeek-V3** (основной), **DeepSeek-Reasoner (R1)** для сложных/длинных текстов.
* **Вход:** `raw_item` на языке оригинала (en, zh, и др.).
* **Выход:** тот же `raw_item` с переведёнными полями `title_ru`, `summary_ru`, `content_ru`.
* **Зачем отдельный агент:** автоматический перевод в составе промпта-автора даёт плохое качество — теряются нюансы, бренды транслитерируются неправильно, рекламная лексика искажается. Отдельный промпт с правилами отрасли даёт качество уровня профессионального переводчика.
* **Важно:** перевод — **рабочий материал** для Author News, а не финальный текст. Author News пишет оригинальную статью с нуля, используя перевод только для извлечения фактов и цифр. Перевод в финальный .md файл не попадает.

**Промпт-каркас:**

```
Ты — профессиональный переводчик материалов о рекламе, маркетинге и adtech.
Переведи текст с {source_language} на русский язык.

Правила:
- Смысловой перевод с адаптацией под СНГ-читателя, НЕ дословный.
- Сохраняй все цифры, факты, имена, даты, ссылки, форматирование markdown.
- Термины adtech — без перевода, если устоялись: DSP, SSP, CPM, CPC, ROAS,
  CTR, ARPU, LTV, RTB, SDK, API.
- Бренды — в устоявшемся русском написании:
  «Яндекс» (не Yandex), «Google», «Alibaba» (не Алибаба), «Baidu» (не Байду),
  «Meta» (запрещён в РФ), «Tencent», «ByteDance», «Ozon», «VK», «2ГИС».
- Имена людей — в устоявшейся транслитерации (Mark Zuckerberg → Марк Цукерберг).
- Если есть ambiguity или игра слов — помечай [translator's note: ...].
- Не добавляй отсебятину, не комментируй, не оценивай.
- Если встречается реалия, непонятная СНГ-читателю (закон, регулирование,
  локальный бренд), добавь краткое пояснение в скобках.

Оригинал:
{content}

Верни только переведённый текст, без пояснений и вступлений.
```

**Тонкости:**

* Длинные тексты (выше 8K токенов) — переключаемся на DeepSeek-Reasoner (R1) для лучшего контекста.
* Уже русскоязычные `raw_item` пропускают этот шаг.
* Промпт валидирован в production-боте (`adtechreview/session_2026_01_23_ideal_format/hybrid_generator.py`), базовые правила переносятся: бренды без транслитерации, термины adtech без перевода, суммы в оригинале.

### 3.4. Filter + Dedup Agent — `processors/filter_dedup.py`

* **Модель:** GLM-4-Flash (embeddings + классификация) + DeepSeek-V3 (семантическая кластеризация).
* **Вход:** пул переведённых `raw_item`-ов.
* **Что делает:**
  1. Повторная фильтрация релевантности уже на русском (после перевода иногда выявляется нерелевантность).
  2. Семантическая дедупликация: embeddings уже опубликованных + кластеризация.
  3. Группирует похожие переведённые raw\_items в «события» (event clusters) с указанием лучшего источника.
* **Выход:** `inbox/curated_items.jsonl` — список уникальных событий на русском с релевантностью и источниками.

### 3.5. Author News — `writers/news_writer.py`

* **Модель:** **DeepSeek-V3** (массовая задача, отличное качество для цены).
* **Эскалация на Claude Sonnet** — для особенно важных/сложных событий (по флагу `priority` или если первый прогон QA FAIL).
* **Вход:** один `curated_item` (переведённые источники для извлечения фактов).
* **Выход:** Markdown-файл с фронт-маттером `type: news`.
* **Частота:** по расписанию, 2 раза в день (10:00 и 18:00 МСК), пул из 5–10 событий за раз.
* **Контроль:** автопубликация (гибридный режим), постфактум-ревью основателем.
* **Критично — осмысленный рерайт, НЕ перевод:** Author News пишет оригинальный текст с нуля. Перевод из Translator — это референс для извлечения фактов и цифр, а не заготовка для правки. Автор получает перевод, но формулировки, структура, угол подачи — свои собственные. Для сверки фактов автору также доступен оригинальный заголовок на английском.

**Промпт-каркас:**

```
Ты — журналист российского профессионального медиа о рекламе и adtech.
Твоя задача — написать оригинальную новость (800–2000 знаков с пробелами)
на основе материалов источников, переведённых на русский.

ВАЖНО: Перевод ниже — рабочий материал для извлечения фактов и цифр.
НЕ копируй формулировки из перевода. Пиши своими словами, как если бы ты
был журналистом, который изучил источники и пишет собственный материал.
Структура, тон, формулировки — твои.

Правила:
- Журналистский, фактологический стиль. Не «пресс-релиз», а новость с контекстом.
- Структура: лид (1–2 предложения) → суть (2–4 абзаца) → «что это значит для
  рекламодателей/рынка» → ссылка на источник.
- Тон: рассуждающий от эксперта, не сухой, но без воды и кликбейта.
- Русский язык, литературная норма. Бренды — в устоявшемся написании.
- Указывай цифры и факты точно — не выдумывай.

КРИТИЧНО — принцип «анти-ИИ-голос» (см. 00_Позиционирование_медиа.md, раздел 7):
- Без ИИ-клише: «почему это важно», «это не X, а Y», «в мире, где...», «далеко
  идущие последствия», «игра меняется», «перепишем правила».
- Без категоричных предсказаний. Прогнозы — как гипотезы с оговорками
  («возможно», «при определённых условиях», «пока рано судить»).
- Признавай unknowns. Если данных недостаточно — говори прямо, не маскируй
  громкими формулировками.
- Допускай сомнение, иронию, оговорки. Не будь «уверенным во всём».

Источники (переведённый рабочий материал):
{sources_block}

Верни результат в формате front-matter + Markdown:
---
title: "..."
description: "..." (до 160 символов)
pubDate: "..."
type: "news"
category: "..."  # одна из: adtech-ru, adtech-world, adtech-asia, market-news, tools, creative
geo: [...]        # массив: ru | world | asia
tags: [...]
source:
  title: "..."
  url: "..."
---

[Тело новости — оригинальный текст, НЕ перевод]
```

### 3.6. Author Digest — `writers/digest_writer.py`

* **Модель:** **DeepSeek-V3** (пользователь подтверждает: отлично справляется с дайджестами и саммаризингом).
* **Вход:** пул всех опубликованных за неделю новостей.
* **Выход:** один Markdown-файл `type: digest`.
* **Частота:** раз в неделю (воскресенье вечером, публикуется в понедельник утром).
* **Контроль:** ревью основателем перед публикацией.

**Промпт-каркас:**

```
Ты — аналитик российского adtech-медиа. Подготовь еженедельный дайджест
по новостям недели.

Вход: список из N новостей за неделю (заголовки + краткое содержание + ссылки).

Структура (проверенный формат из production-бота):

1. **Главная новость недели** — одно событие, которое выходит за рамки
   отраслевых новостей. Заголовок + 2-3 предложения описания + 1 наблюдение.
2. **В фокусе** — 2-3 значимые новости, каждая с коротким комментарием
   (1-2 абзаца).
3. **Кратко** — 5-8 новостей сгруппированных по темам:
   - Retail Media
   - AI & Automation
   - Measurement & Privacy
   - Платформы (Google, Meta (запрещён в РФ), Amazon, ...)
   - M&A и инвестиции
   Каждая — заголовок + 1 предложение + ссылка.
4. **Что это значит для рынка** — 2-3 наблюдения с оговорками
   («возможно», «при определённых условиях», «пока рано судить»).

Объём: 800–1500 слов. Тон: экспертно-аналитический, журналистский.
Короткие абзацы по 1-2 предложения. Допустима лёгкая ирония.
Меньше глобальных выводов — наблюдения, а не прогнозы.

Принцип «анти-ИИ-голос» — обязательно (см. 00_Позиционирование_медиа.md, раздел 7).

[Список новостей]

Верни результат с фронт-маттером type: digest.
```

> **Источник формата:** `adtechreview/session_2026_01_23_ideal_format/ideal_digest_format.md` — структура отшлифована за месяцы работы бота.

### 3.7. Author Review — `writers/review_writer.py`

* **Модель:** **Claude Sonnet** (здесь качество важнее стоимости, материалов меньше — 1–3 в неделю).
* **Вход:** конкретный инструмент / продукт + исследованный материал (web search или собранные данные).
* **Выход:** Markdown-файл `type: review`.
* **Частота:** 1–3 в неделю.
* **Контроль:** ревью основателем.

**Промпт-каркас:**

```
Ты — эксперт-обозреватель рекламных и маркетинговых инструментов.
Напиши обзор инструмента "{tool_name}" для российских маркетологов.

Структура:
1. Что за инструмент, кому нужен (1–2 абзаца).
2. Возможности (что умеет, форматы, интеграции).
3. Цены (если есть данные).
4. Плюсы и минусы (по 3–5 пунктов, честно).
5. Кому подойдёт, кому нет.
6. Вердикт.

Объём: 1000–2500 слов. Тон: практический, прикладной, с конкретикой.
Не хвалить безосновательно. Если минусы — называть.
Принцип «анти-ИИ-голос» — обязательно.

Исходные данные:
{research_data}

Верни результат с фронт-маттером type: review, category: tools.
```

### 3.8. Author Reports — `writers/reports_writer.py`

* **Модель:** **Claude Opus** для research и структуры + **Claude Sonnet** для написания (многоэтапный процесс).
* **Вход:** тема отчёта + собранные данные (финансовые отчёты, опросы, рыночные данные).
* **Выход:** Markdown-файл `type: report`.
* **Частота:** 1–2 в месяц + квартальные «состояние рынка».
* **Контроль:** всегда ручной, многоэтапный (структура → данные → факты → выводы).
* **Workflow (несколько прогонов):**
  1. Opus готовит структуру и outline.
  2. Человек ревьюит структуру.
  3. Opus/Sonnet собирает данные по разделам.
  4. Sonnet пишет каждый раздел отдельно.
  5. Человек ревьюит и редактирует.
  6. Финальная сборка.

**Промпт-каркас (основной):**

```
Ты — аналитик-исследователь российского медиа о рекламе и adtech.
Готовишь глубокий отраслевой отчёт: "{topic}".

Объём: 2500–8000 слов. Структура из outline (см. ниже).

Обязательные элементы:
- Описание методологии (источники данных, период, ограничения).
- Графики и таблицы (не описывать цифры в тексте — оформлять как таблицу).
- Раздел «Что это значит для рынка» — выводы с явными оговорками.
- Confidence-уровни по прогнозам.
- Раздел «Что мы НЕ знаем» — честно об unknowns.

Тон: экспертно-аналитический, с сомнением и оговорками.
Принцип «анти-ИИ-голос» — обязательно.

Outline:
{outline}

Данные:
{research_data}

Верни результат с фронт-маттером type: report.
```

### 3.9. Enricher (SEO + Geo + теги) — `processors/enricher.py`

* **Модель:** **GLM-4-Flash** (бесплатно, много вызовов, простая задача).
* **Вход:** черновик Markdown.
* **Что делает:**
  1. Проверяет и дополняет теги (из контролируемого словаря).
  2. Уточняет категорию и гео-пометку.
  3. Генерирует SEO-friendly slug для URL.
  4. Генерирует `description` (160 символов), если нет.
  5. Предлагает заголовок для соцсетей (короткий, твит-формат).
  6. Проверяет соответствие чек-листу публикации (см. `06_SEO_GEO_и_AI-разметка.md`, раздел 11).
* **Выход:** тот же Markdown с дополненным фронт-маттером.
* **Системный промпт** формируется на основе `06_SEO_GEO_и_AI-разметка.md`.

### 3.10. QA / Validator — `processors/qa.py`

* **Модель:** правило-based (jsonschema, длины, slug-проверка) + **GLM-4-Flash** для мягкой проверки фактов.
* **Что проверяет:**
  1. Корректность схемы фронт-маттера (через Astro Content Collections schema).
  2. Длина материала (min/max для каждого типа).
  3. Уникальность slug.
  4. Наличие всех обязательных полей.
  5. Наличие ссылки на источник.
  6. Отсутствие явных галлюцинаций (ключевые цифры из сырья должны быть в материале).
  7. Анти-паттерны — проверка на запрещённые ИИ-клише через regex + LLM.
* **Выход:** PASS / FAIL с причиной. FAIL — откат, алерт.

### 3.11. Publisher — `publishers/git_publisher.py`

* **Не LLM-агент**, а Python-скрипт.
* **Что делает:**
  1. Читает черновик из `drafts/`.
  2. Если QA PASS и `type in ("news",)`: коммитит напрямую в `main` ветку `media-site` с сообщением `news: [slug]`.
  3. Если `type in ("digest", "review", "report")`: создаёт PR в `media-site` с веткой `draft/[slug]`, ставит в описание сырьё и логику автора.
  4. Удаляет черновик из `drafts/`.
  5. Логирует в `data/published.jsonl`.

| Этап               | Модель                    | Почему именно она                                                 | Стоимость/мес (оценка)     |

## 5. Workflow по типам контента

### 5.1. Короткая новость (полностью авто)

```
RSS/блог
  → raw_items (на en/zh)
  → pre_filter (GLM-Flash)
  → translator (DeepSeek-V3) → на русском
  → filter_dedup
  → news_writer (DeepSeek-V3)
  → enricher (GLM-Flash)
  → qa (rules + GLM-Flash)
  → publisher (push main) → Timeweb
```

Время от сбора до публикации: 45–90 минут (дольше на 15-30 минут из-за перевода).

### 5.2. Дайджест недели (с ревью)

```
published news → digest_writer (DeepSeek-V3, воскресенье)
  → enricher → qa → publisher (PR) → ревью → merge → Timeweb
```

### 5.3. Обзор инструмента (с research)

```
owner → запрос на обзор tool_X
  → researcher собирает данные (DeepSeek-V3 / web search)
  → review_writer (Claude Sonnet)
  → enricher → qa → publisher (PR) → ревью → merge → Timeweb
```

### 5.4. Отчёт/исследование (многоэтапный, человек-heavy)

```
owner → тема + outline (Claude Opus)
  → человек ревьюит outline
  → research по разделам (Claude Opus)
  → написание разделов (Claude Sonnet)
  → человек редактирует
  → финальная сборка → publisher (PR) → ревью → merge → Timeweb
```

### 5.5. Экспертная колонка (только человек)

```
owner пишет в Markdown → push в src/content/columns/ → Timeweb
```

Агенты НЕ касаются колонок. Опционально: Enricher может проверить SEO после пуша.

## 6. Контроль качества

### На уровне pre-filter

* Скор релевантности ≥ 5/10.

### На уровне фильтрации после перевода

* Повторная оценка релевантности уже на русском.
* Не дубликат уже опубликованного (cosine similarity \< 0.85).

### На уровне автора

* Минимальная длина: 500 знаков с пробелами (для новостей). Всё короче — брак.
* Ориентир: 500–1000 знаков. Сверх — нормально, если контекст оправдан.
* Наличие лида, контекста, ссылки на источник.

### На уровне QA

* Соответствие схеме Astro Content Collections.
* Slug уникальный.
* Title 50–80 символов.
* Description ≤ 160 символов.
* Контроль фактов: ключевые цифры из сырья должны быть в материале.
* **Анти-ИИ-паттерны:** regex + LLM-проверка на запрещённые клише из принципа «анти-ИИ-голос».

### Постфактум-ревью основателем

* Каждый день: беглый просмотр опубликованных новостей.
* Если качество плохое — корректировка промпта в AI sreda + перепубликация.
* При повторяющихся FAIL на DeepSeek — эскалация конкретного типа материалов на Claude Sonnet.

## 7. Обработка ошибок

| Случай                                                     | Поведение                                                                  |
| ---------------------------------------------------------- | -------------------------------------------------------------------------- |
| RSS-фид недоступен                                         | Пропуск с warning в лог, не падать                                         |
| Превышение лимита API (любого провайдера)                  | Pause + retry с экспоненциальной задержкой                                 |
| DeepSeek API недоступен                                    | Fallback на Claude Sonnet для критичных материалов; пауза для массовых     |
| GLM API недоступен                                         | Pre-filter и enricher пропускаются (с алертом), pipeline продолжает работу |
| LLM вернул невалидный Markdown                             | QA FAIL, черновик уходит в `drafts/failed/`, алерт                         |
| Git push отклонён                                          | Откат, алерт                                                               |
| Дубликат обнаружен постфактум                              | Удалить файл, перекоммитить с пометкой                                     |
| Перевод явно плохой (определяется QA по русским паттернам) | Повторный прогон Translator с упрощённым промптом                          |

Все алерты — в Telegram-бот через Bot API.

## 8. Метрики работы пайплайна

Замеряем в `state/metrics.jsonl` и `state/llm_usage.jsonl`:

**Производительность:**

* Количество собранных raw\_items в день.
* Доля прошедших pre-filter.
* Доля прошедших финальный filter.
* Количество опубликованных материалов в день (по типам).
* Доля QA FAIL (по этапам и моделям — для оптимизации).
* Среднее время от сбора до публикации.

**Экономика LLM:**

* Токены in/out по каждой модели (DeepSeek, GLM, Claude).
* Стоимость по каждой модели в день/неделю/месяц.
* Стоимость в пересчёте на опубликованный материал.
* Таргет: **$0.30–0.50 на материал** на старте.

**Качество (после первых 30 публикаций):**

* Доля материалов, потребовавших ручной правки.
* Доля эскалаций DeepSeek → Claude.
* Срабатывания анти-ИИ-паттернов в QA.

## 9. Что реализуется в Claude Code

На основе этого документа:

1. **Скелет проекта** (структура каталогов, `requirements.txt`, `.env.example`).
2. **Коллекторы** (начать с RSS — самый простой).
3. **Pre-filter** (GLM-4-Flash + JSON-парсинг ответа).
4. **Translator** (DeepSeek API, отдельный промпт, кэш переводов).
5. **Filter + Dedup** (Python + embeddings через GLM).
6. **Агенты-авторы** (мультипровайдерный клиент: Anthropic SDK + DeepSeek SDK + GLM SDK).
7. **Enricher и QA** (rules + LLM).
8. **Publisher** (GitPython или subprocess + git).
9. **GitHub Actions workflows** (`.github/workflows/`).
10. **Дашборд стоимости** — простой скрипт, читает `llm_usage.jsonl` и выводит недельный отчёт.

**Абстракция LLM-провайдера:** общий интерфейс `LLMClient` с реализациями `DeepSeekClient`, `GLMClient`, `ClaudeClient`. Это позволит легко добавлять новые модели или менять распределение.

## 10. Что ведётся в AI sreda

* **Промпт-инжиниринг:** итеративная работа над промптами каждого агента (хранятся в `media-agents/src/prompts/*.md` или прямо здесь, в проекте).
* **Контент-план:** темы для обзоров, колонок, дайджестов, отчётов.
* **Ревью материалов:** сюда падают черновики перед публикацией (через интеграцию; на старте — копипастом).
* **Мониторинг экономики:** регулярный анализ `llm_usage.jsonl`, корректировка распределения моделей.
* **Стратегические итерации:** пересмотр рубрикатора, тональности, источников, эскалаций.
