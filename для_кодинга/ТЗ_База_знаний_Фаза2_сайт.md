## 0. Что эмитит backend (вход для сайта)

Файлы `media-site/src/content/spravochnik/{slug}.md`. Front-matter (из `src/spravochnik/writer.py`):

```yaml
title: "Adobe"
description: "…до 160 символов…"
type: company            # company | technology | term | organization
slug: adobe              # = имя файла; в схеме — optional (или дропнуть, см. §2 прим.)
pubDate: 2026-07-19T12:00:00Z   # ISO, utcnow (как у новостей)
aliases: ["Adobe Inc."]  # опц.
tags: [adtech, platform, software]   # 3–7
related: [criteo, dv360] # опц., slug'и других материалов справочника
facts:                   # состав зависит от type — см. §2
  founded: 1982
  founders: ["Джон Уорнок", "Чарльз Гешке"]
  hq: "Сан-Хосе, Калифорния, США"
  official_url: "https://www.adobe.com/"
  key_products: ["Adobe Advertising Cloud", "…"]
# author — НЕ эмитится; updatedDate — опц. (задел под update-flow)
```

Тело — markdown по структуре типа (§ разделы из ТЗ раздела 2). JSON-LD в файле НЕТ.

Чтобы совпасть с конвенциями сайта, сначала прочитать:

3. `media-site/src/layouts/*` (BaseHead/Layout) — meta/OG/canonical/Twitter, как страница

## 2. Коллекция `spravochnik` в `content.config.ts`

Добавить коллекцию с **discriminated union по `type`** (4 набора `facts`). ⚠️Синтаксис loader'а
скопировать у существующих коллекций (glob vs legacy).

```ts
// общие поля всех типов
const spravBase = {
  title: z.string(),
  description: z.string().max(160),
  slug: z.string().optional(),          // прим.: Astro берёт id из имени файла; поле терпим
  pubDate: z.coerce.date(),
  updatedDate: z.coerce.date().optional(),
  aliases: z.array(z.string()).default([]),
  tags: z.array(z.string()).default([]),
  related: z.array(z.string()).default([]),
};

const spravochnik = defineCollection({
  // ⚠️loader/type — как у news в этом же файле:
  loader: glob({ pattern: '**/*.md', base: './src/content/spravochnik' }),
  schema: z.discriminatedUnion('type', [
    z.object({ ...spravBase, type: z.literal('company'),
      facts: z.object({
        founded: z.union([z.number(), z.string()]),
        founders: z.array(z.string()),
        hq: z.string(),
        official_url: z.string().url(),
        subtype: z.string().optional(),
        parent_organization: z.string().nullable().optional(),
        ticker: z.string().optional(),
        key_products: z.array(z.string()).default([]),
      }) }),
    z.object({ ...spravBase, type: z.literal('technology'),
      facts: z.object({
        developer: z.string(),
        category: z.string(),
        official_url: z.string().url(),
        launch_date: z.union([z.number(), z.string()]).optional(),
        pricing_model: z.string().optional(),
        alternatives: z.array(z.string()).default([]),
      }) }),
    z.object({ ...spravBase, type: z.literal('term'),
      facts: z.object({
        category: z.string(),
        definition: z.string(),
        aliases: z.array(z.string()).default([]),
      }) }),
    z.object({ ...spravBase, type: z.literal('organization'),
      facts: z.object({
        full_name: z.string(),
        founded: z.union([z.number(), z.string()]),
        mission: z.string(),
        official_url: z.string().url(),
        key_initiatives: z.array(z.string()).default([]),
      }) }),
  ]),
});

// не забыть добавить spravochnik в export const collections = { … }
```

**⚠️Важно — `NO author`:** НЕ добавлять `author: reference('authors')` в схему справочника
(владелец: автор не нужен; backend его не эмитит).

**Прим. про `slug`:** backend пишет `slug` во front-matter, но роут/id Astro берёт из имени файла.
Схема помечает `slug` optional (терпим). Если существующие коллекции сайта используют `z.object().strict()`
или иначе ругаются на лишние поля — либо оставить `slug` в схеме (как выше), либо убрать эмиссию
`slug` из `writer._finalize_meta` в media-agents. По умолчанию — оставить optional.

**⚠️description.max(160):** backend уже режет до 160 (`_fit_description`) — схема совпадает.
Если у сайта строже (напр. min) — сверить.

## 3. JSON-LD — генерация в Astro (решение A)

Компонент `media-site/src/components/SpravochnikJsonLd.astro` (или inline в layout), `switch (type)`:

```astro
---
const { entry, site } = Astro.props;   // entry = запись коллекции; site = import.meta.env.SITE
const f = entry.data.facts;
const url = new URL(`/spravochnik/${entry.id}`, site).href;
const dateModified = (entry.data.updatedDate ?? entry.data.pubDate).toISOString();

let node;
if (entry.data.type === 'company' || entry.data.type === 'organization') {
  node = {
    '@type': 'Organization',
    name: entry.data.title,
    ...(entry.data.aliases?.length ? { alternateName: entry.data.aliases } : {}),
    ...(f.founded ? { foundingDate: String(f.founded) } : {}),
    ...(f.founders ? { founders: f.founders.map((n) => ({ '@type': 'Person', name: n })) } : {}),
    ...(f.hq ? { address: { '@type': 'PostalAddress', addressLocality: f.hq } } : {}),
    ...(f.ticker ? { tickerSymbol: f.ticker } : {}),
    url: f.official_url,
  };
} else if (entry.data.type === 'technology') {
  node = {
    '@type': 'SoftwareApplication',
    name: entry.data.title,
    applicationCategory: 'AdvertisingApplication',
    operatingSystem: 'Web',
    ...(f.developer ? { creator: { '@type': 'Organization', name: f.developer } } : {}),
    ...(f.launch_date ? { releaseDate: String(f.launch_date) } : {}),
    url: f.official_url,
  };
} else { // term
  node = {
    '@type': 'DefinedTerm',
    name: entry.data.title,
    description: entry.data.description,
    inDefinedTermSet: { '@type': 'DefinedTermSet', name: 'Справочник 1screen' },
  };
}
const jsonld = { '@context': 'https://schema.org', ...node, dateModified };

// хлебные крошки
const breadcrumbs = {
  '@context': 'https://schema.org', '@type': 'BreadcrumbList',
  itemListElement: [
    { '@type': 'ListItem', position: 1, name: 'Главная', item: new URL('/', site).href },
    { '@type': 'ListItem', position: 2, name: 'База знаний', item: new URL('/spravochnik/', site).href },
    { '@type': 'ListItem', position: 3, name: entry.data.title, item: url },
  ],
};
---
<script type="application/ld+json" set:html={JSON.stringify(jsonld)} />
<script type="application/ld+json" set:html={JSON.stringify(breadcrumbs)} />
```

⚠️Свериться с тем, как `article/[...slug].astro` уже вставляет JSON-LD (set:html / компонент) —
использовать тот же приём и тот же `site`-URL.

## 4. Роуты

### `media-site/src/pages/spravochnik/[slug].astro`
- `getStaticPaths` — по всем записям коллекции `spravochnik` (⚠️API: `getCollection('spravochnik')`).
- Рендер: заголовок + **бейдж типа** (Компания/Технология/Термин/Организация), карточка `facts`
  (основана/основатели/HQ/… по типу), тело (`<Content />`), блок «Связанные термины» (§5),
  блок «Читайте также» (связанные новости — ⚠️опц., только если позже прокинем из researcher;
  на старте можно опустить). В `<head>` — `SpravochnikJsonLd` + meta/OG/canonical.

### `media-site/src/pages/spravochnik/index.astro`
- Индекс всех материалов, **сгруппированных по типу** (Компании / Технологии / Термины /
  Организации) — краулабельно, без query-параметров (лучше для SEO, см. `06_SEO_GEO` §8;
  `?type=` даёт `noindex`). Сортировка внутри группы по `pubDate` убыв.
- ⚠️Клиентский фильтр `?type=` можно добавить поверх (прогрессивно), но базовый индекс —
  статический со всеми секциями.

## 5. Фильтр `related` (решение C — без 404)

В `[slug].astro` линковать только СУЩЕСТВУЮЩИЕ slug'и:

```astro
const all = await getCollection('spravochnik');
const ids = new Set(all.map((e) => e.id));
const relatedEntries = (entry.data.related ?? [])
  .filter((s) => ids.has(s))
  .map((s) => all.find((e) => e.id === s));
// рендерить relatedEntries → <a href={`/spravochnik/${e.id}`}>{e.data.title}</a>
```

Несуществующие `related` молча отбрасываются — битых внутренних ссылок нет.
