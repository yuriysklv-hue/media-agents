## ТЗ (реализовано 18.07.2026 — см. секцию выше): Дайджест недели на материалах сайта (задача №3)

> Решение владельца (18.07.2026). ТЗ ниже реализовано в сессии 18.07 — оставлено как эталон.

### Проблема и решение

Дайджест **уже не пересобирается из RSS** — `digest_writer._news_for_week` читает
`published.jsonl` (`state.load_published()`) и фильтрует по неделе. Но `published.jsonl`
= «**отправлено в PR**», а НЕ «**живёт на сайте**»:
- `pr_merged` пишется `False` в `git_publisher._record_published` и **никогда не
  обновляется** в `True` (проверено: единственное вхождение — запись `False`);
- владелец **вручную удалял** часть материалов из PR перед мержем (дубли и др.) →
  такие статьи остаются в `published.jsonl`, но на сайте их нет → дайджест дал бы
  **битые ссылки** (`/article/{slug}` в никуда).

**Решение — источник истины = реальная коллекция сайта в репо `media`**
(`media-site/src/content/news/*.md`, только то, что смёржено и живёт). `published.jsonl`
из дайджеста **убрать**. **Фолбэка НЕТ:** если клон `media` не поднялся или за неделю
нет статей — дайджест НЕ выпускать (падать/None). Логика владельца: *лучше нет
дайджеста, чем дайджест с битыми ссылками*. Локальных прогонов дайджеста не будет,
поэтому городить откат на `published.jsonl` не нужно.

`published.jsonl` НЕ трогаем — он остаётся под сверку «отправлено ↔ на сайте» (задача №7).

### Почему это реально (доступ к `media` из `media-agents` уже есть)

- `digest.yml` уже прокидывает в `run_digest.py` секреты `GH_TOKEN` (PAT на `media`),
  `MEDIA_REPO`, `MEDIA_SITE_SUBDIR` — те же, что у паблишера.
- `git_publisher._prepare_clone(git_url)` уже клонирует/обновляет `media` в
  `CLONE_DIR = ROOT/media-clone` (main, depth 1), а `_git_env()` собирает
  авторизованный URL из `GH_TOKEN`+`MEDIA_REPO`. Паблишер уже читает/пишет по пути
  `CLONE_DIR/{MEDIA_SITE_SUBDIR}/src/content/{news|digest}/` — **ровно та папка**,
  где лежат живые новости. Дайджест читает тот же клон, просто раньше и на чтение.

### План реализации

1. **Общий хелпер клона `media`** — вынести `_git_env()` + `_prepare_clone()` из
   `src/publishers/git_publisher.py` в новый модуль `src/publishers/media_repo.py`
   (паблишер импортирует оттуда, поведение НЕ менять — обратная совместимость).
   Экспорт:
   - `clone_or_update_media() -> Path` — путь к клону `media` на свежем `main`;
   - `news_dir(clone: Path) -> Path` = `clone / MEDIA_SITE_SUBDIR / "src/content/news"`.
2. **`src/writers/digest_writer.py`:**
   - Заменить `_news_for_week(state, week)` → `_news_for_week_from_site(week)`:
     - `clone_or_update_media()` → прочитать все `.md` в `news_dir`;
     - для каждого: `split_front_matter` (уже есть), **slug = имя файла** (`path.stem`);
     - фильтр: `pubDate` (date-часть) в границах `week_bounds(week)` включительно;
     - собрать запись `{slug, title, category, geo, source_name(=source.title),
       source_url(=source.url), pub_date}`; сортировать по `pub_date`.

Из-за этого на сайте статьи датировались задним числом (напр. 07-08 вместо реальной публикации 07-09): выглядели несвежими, не зажигали live-точку на главной (`isLive` = моложе 3 ч), а батчи со старыми датами источника **тонули внизу ленты** (сортировка по `pubDate` убыв.).

**Фикс:** детерминированная доводка front-matter вынесена в `news_writer._finalize_meta`, где `pubDate = utcnow_iso()` (момент выхода на 1screen). Дата источника не теряется — паблишер пишет её в `published.jsonl` отдельным полем `source_published_at` (`src/publishers/git_publisher.py`); во front-matter она не идёт, чтобы не расширять контракт схемы сайта. QA-проверка «pubDate в будущем» проходит (дата всегда в прошлом). Промпт `news_writer.md` — `pubDate: "AUTO"` (значение всё равно перетирается кодом). Тесты: `tests/test_content_feedback.py` (`test_pubdate_is_now_not_source_date`, `test_pubdate_not_in_future_for_qa`, `test_finalize_sets_source_and_defaults`).

## Задача 2: механизм корпуса «качественные / некачественные»

Владельцу нужно докручивать агента написания на реальных примерах. Сейчас:
- **хороший корпус** = опубликованные статьи (в репо `media`, `media-site/src/content/news/`);
- **плохой корпус** = забракованные QA черновики уходят в `drafts/failed/` **на эфемерном раннере и теряются** — их не видно.

Надо **сохранять забракованные черновики + вердикт QA (с причиной)** между прогонами: либо коммитить `data/drafts/failed/` в репо, либо загружать как artifact воркфлоу (`actions/upload-artifact`). Тогда у владельца будет размеченный набор «прошло / не прошло + почему» для тюнинга промптов. Причины QA уже пишутся в лог — можно дополнительно собирать их в сводный отчёт (slug → вердикт → причина).

## Задача 4/5 (баг, ✅ ЗАКРЫТО 13.07.2026): двоеточие в заголовке ломает front-matter

Один материал упал: `mapping values are not allowed here` — DeepSeek поставил `:` в значение YAML без кавычек, весь материал терялся. **Фикс:** `split_front_matter` (`src/utils/frontmatter.py`) обёрнут в try — при `yaml.YAMLError` запускается `_repair_front_matter`, который квотит незакавыченные значения строковых полей (`title`, `description`, `social_title`, `pubDate`) и парсит YAML повторно. Штатный (валидный) front-matter не затрагивается — починка срабатывает только на ошибке. Промпт `news_writer.md` дополнительно просит ставить двоеточие только внутри кавычек. ## Задача 7 (БАГ, ✅ ЗАКРЫТО 13.07.2026): «Commit data state» падал с 403 → состояние дедупа не сохранялось

**Решено (PR #16):** в `news-pipeline.yml` и `digest.yml` добавлен блок `permissions: contents: write`. Оказалось достаточно **только** этого — настройку репо (Settings → Actions → Workflow permissions) владелец не трогал, дефолт токена остался read-only, но блок `permissions:` на уровне воркфлоу его переопределяет. Проверено боем: digest run 13.07 — «Commit data state» зелёный, `git push` без 403. Вариант с PAT (ниже) не понадобился. Ниже — исходный разбор, оставлен для истории.

Финальный шаг воркфлоу `git push` падает с **403 `Permission to …/media-agents.git denied to github-actions[bot]`** — у дефолтного `GITHUB_TOKEN` нет прав на запись (в `news-pipeline.yml` и `digest.yml` нет блока `permissions`, а в настройках репо Actions, судя по всему, стоят read-only). Job краснеет уже **после** создания PR — PR-ы идут через отдельный PAT `GH_TOKEN`, поэтому проходят, а вот коммит состояния пушится дефолтным токеном и отлетает.

**Прошлый диагноз был неверен.** Дело НЕ в add/add-конфликте `git pull --rebase origin main` на фиче-ветке. Проверено на боевом прогоне 09.07 (run `29008723017`, ветка **`main`**): rebase прошёл чисто (`Current branch main is up to date`), упал именно `git push` с 403. Значит и на `main` шаг НЕ проходит — `data/state/` и `data/published/` не коммитятся **никогда**.

**Почему это важно (а не косметика):**
- `data/state/seen_urls.json` и `translation_cache.json` не сохраняются → память дедупа обнуляется каждый прогон → прямой вклад в дубли (**Задача 6**; 09.07 поймали двойной DuckDuckGo).
- `data/published/published.jsonl` не сохраняется → ломает сверку «отправлено ↔ на сайте» (**Задача 1**).
- Каждый прогон помечается красным, хотя контент и PR делаются — маскирует настоящие падения.

**Решение (достаточно любого варианта):**
1. **Дать воркфлоу права на запись (проще, предпочтительно).** Добавить в `news-pipeline.yml` и `digest.yml`:
   ```yaml
   permissions:
     contents: write
   ```
   ~~и в Settings → Actions → General → Workflow permissions выставить «Read and write permissions».~~ (Не понадобилось — блока `permissions:` в воркфлоу достаточно, репо-настройку не меняли.)
2. **Либо пушить состояние через PAT.** В шаге «Commit data state» пушить не дефолтным токеном, а токеном с write-доступом на `media-agents` (`git push https://x-access-token:${TOKEN}@github.com/<owner>/media-agents`). Учесть: текущий `GH_TOKEN` заведён под создание PR в репо `media` — для этого варианта у него должен быть `repo`-скоуп и на `media-agents`.

После фикса убедиться, что `data/state/` реально обновляется в `main` после прогона (по коммиту `state: pipeline run …`), и что дубли из Задачи 6 перестают протекать между прогонами.

> ⚠️ Тот же устаревший диагноз («add/add-конфликт с фиче-ветки») продублирован в `CLAUDE.md` (раздел «Подводные камни») — при фиксе поправить и там.
